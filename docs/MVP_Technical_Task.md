# Техническое задание: Этап 1 — MVP (Минимально жизнеспособный продукт)

## 1. Цель этапа
Запуск базовой версии платформы **EduAI**, предоставляющей пользователям (студентам и преподавателям) доступ к ключевым функциям через Telegram-бота:
- Поиск научной литературы (PsyArXiv).
- Ответы на вопросы по загруженным документам (RAG).
- Базовая проверка домашних заданий.
- Единый интерфейс управления через чат-бота.

**Срок реализации:** 4-6 недель.
**Критерий успеха:** Работающий бот в тестовом контуре, способный обработать запросы от 10 одновременных пользователей без критических ошибок.

---

## 2. Архитектурные задачи (Infrastructure & Core)

### 2.1. Настройка окружения и репозитория
- [ ] **Инициализация проекта:** Создание структуры папок (`core`, `agents`, `services`, `tests`, `docs`).
- [ ] **Конфигурация:** Настройка `.env` для хранения ключей API (Cloud.ru, Telegram, LLM).
- [ ] **Контейнеризация:** Написание `Dockerfile` и `docker-compose.yml` для поднятия сервиса бота и зависимостей.
- [ ] **Логирование:** Внедрение структурированного логирования (JSON format) с уровнями INFO, WARNING, ERROR.

### 2.2. Ядро системы (Main Agent Orchestrator)
- [ ] **Роутинг запросов:** Реализация логики определения интента пользователя (Поиск статьи / Вопрос по документу / Проверка ДЗ).
- [ ] **Управление контекстом:** Разработка механизма хранения краткосрочной памяти диалога (до 10 сообщений) в Redis или in-memory (для MVP).
- [ ] **Обработка ошибок:** Глобальный перехват исключений и формирование понятных ответов пользователю при сбоях API.

### 2.3. Интеграция с Telegram
- [ ] **Базовый бот:** Подключение `python-telegram-bot` (v20+), реализация команд `/start`, `/help`, `/settings`.
- [ ] **Асинхронность:** Обеспечение неблокирующей обработки запросов (использование `asyncio`).
- [ ] **Форматирование:** Поддержка Markdown/HTML в ответах бота (жирный текст, списки, ссылки).

---

## 3. Функциональные задачи по модулям

### 3.1. Модуль «Поиск литературы» (PsyArXiv Selector) — *Приоритет: Высокий*
*Цель: Поиск статей по ключевым словам в открытых репозиториях.*

- [ ] **API Интеграция:** Написание клиента для работы с API OSF/PsyArXiv.
- [ ] **Фильтрация:** Реализация фильтрации результатов по дате и релевантности (топ-5 выдачи).
- [ ] **Формирование ответа:** Генерация карточки статьи: Заголовок, Авторы, Краткое описание (Abstract), Ссылка на PDF.
- [ ] **Команда бота:** Реализация команды `/search <запрос>`.

### 3.2. Модуль «Вопрос-ответ по документам» (RAG Service) — *Приоритет: Высокий*
*Цель: Загрузка файла пользователем и получение ответов по его содержимому.*

- [ ] **Загрузка файлов:** Реализация приема файлов (PDF, TXT) от пользователя в чате (лимит до 10 Мб).
- [ ] **Обработка текста:** Парсинг текста из файлов (библиотеки `PyPDF2` или `pdfplumber`).
- [ ] **Векторизация (MVP):** Разбивка текста на чанки и использование простой эмбеддинг-модели (локально или через API Cloud.ru).
- [ ] **Поиск и генерация:** Поиск релевантных чанков по вопросу пользователя и формирование ответа через LLM.
- [ ] **Ограничение:** Для MVP хранение векторов только в рамках одной сессии (без постоянной БД векторов).

### 3.3. Модуль «Проверка домашних заданий» (Homework Checker) — *Приоритет: Средний*
*Цель: Предварительный анализ текста студенческой работы.*

- [ ] **Шаблон промпта:** Разработка системного промпта для роли "Строгий преподаватель".
- [ ] **Критерии проверки:** Реализация анализа по параметрам: Грамотность, Структура, Соответствие теме, Наличие выводов.
- [ ] **Формат отчета:** Генерация структурированного ответа: Оценка (1-10), Список замечаний, Рекомендации по улучшению.
- [ ] **Команда бота:** Реализация команды `/check` (ответ на файл или текст).

### 3.4. Модуль «Помощник преподавателя» (Teacher Assistant) — *Приоритет: Низкий (базовая версия)*
*Цель: Доступ к расписанию и статике.*

- [ ] **Статичное расписание:** Возможность загрузить расписание (JSON/CSV) администратором и выводить его по команде `/schedule`.
- [ ] **FAQ:** База частых вопросов и ответов для студентов (реализация через простой поиск по ключевым словам).

## 4. Требования к качеству и тестированию

### 4.1. Юнит-тесты
- [ ] Покрытие тестами основных функций парсинга и роутинга (минимум 60% покрытия).
- [ ] Моки (mocks) для внешних API (Telegram, LLM, PsyArXiv) для тестирования без реальных вызовов.

### 4.2. Интеграционное тестирование
- [ ] Сценарий "Happy Path": Пользователь отправляет файл -> Бот обрабатывает -> Возвращает ответ.
- [ ] Сценарий "Error Handling": Отправка неподдерживаемого формата файла -> Корректное сообщение об ошибке.

### 4.3. Нагрузочное тестирование (Базовое)
- [ ] Проверка работы бота при 5 одновременных запросах (отсутствие таймаутов более 30 сек).

---

## 5. План сдачи этапа (Deliverables)

1. **Исходный код:** Ветка `main` с работающим кодом всех модулей MVP.
2. **Документация:**
   - Инструкция по развертыванию (`README.md`).
   - Описание переменных окружения (`.env.example`).
   - Примеры диалогов с ботом.
3. **Демо-стенд:** Развернутый экземпляр бота в Telegram для демонстрации заказчику.
4. **Отчет о тестировании:** Краткий список известных ограничений и багов (Backlog для Этапа 2).

---

## 6. Риски и ограничения MVP

| Риск | Вероятность | Митигация (Снижение риска) |
|------|-------------|----------------------------|
| Лимиты API LLM | Высокая | Кэширование ответов, ограничение длины контекста |
| Качество парсинга PDF | Средняя | Поддержка только текстовых PDF, предупреждение пользователя о сканах |
| Таймауты Telegram | Средняя | Уведомление пользователя "Думаю..." (sendChatAction) при долгих запросах |
| Точность RAG | Средняя | Добавление дисклеймера "Ответ ИИ может содержать ошибки" |
